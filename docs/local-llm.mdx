## Why We Chose Cloud-Only

OCA now targets cloud-hosted LLMs because Android devices are constrained by memory, storage, thermal limits, and CPU-only inference speed. The table below captures the practical reasons local model support was removed.

| Constraint | Details |
|------------|---------|
| **RAM** | GGUF models need at least **2-4GB of free memory** for small 7B-class Q4 models, while phone RAM is shared with Android and other apps |
| **Storage** | Model files range from **4GB to 70GB+**, which can quickly consume phone storage |
| **Speed** | **CPU-only inference** on ARM is very slow, and Android does not support practical GPU offloading for llama.cpp in OCA |
| **Use Case** | OpenClaw routes to **cloud LLM APIs** such as OpenAI, Gemini, Ollama Cloud, and other hosted providers for production-speed responses |

<Info>
  Cloud-hosted models avoid local model downloads, reduce RAM pressure, and keep OCA behavior consistent across Android devices.
</Info>

## Ollama Cloud Models

Ollama Cloud Models are the recommended Ollama path for OCA on Android. They keep the familiar `ollama` workflow while running inference on cloud infrastructure instead of the phone.

### Quick Start

```bash
# Pull and launch with a cloud model
ollama pull kimi-k2.5:cloud
ollama launch openclaw --model kimi-k2.5:cloud
```

### Recommended Cloud Models

| Model | Use Case | Context |
|-------|----------|---------|
| **`kimi-k2.5:cloud`** | Multimodal reasoning with subagents | 64k+ tokens |
| **`minimax-m2.5:cloud`** | Fast, efficient coding | 64k+ tokens |
| **`glm-5:cloud`** | Reasoning and code generation | 64k+ tokens |
| **`gpt-oss:120b-cloud`** | High-performance tasks | 128k tokens |
| **`gpt-oss:20b`** | Balanced performance | 64k tokens |

### Commands

| Command | Description |
|---------|-------------|
| `ollama launch openclaw` | Launch with the model selector |
| `ollama launch openclaw --model <model>` | Launch with a specific cloud model |
| `ollama launch openclaw --config` | Configure without launching |
| `ollama pull <model>:cloud` | Pull a cloud model reference into the local registry |

### Why Cloud Models?

| Advantage | Details |
|-----------|---------|
| **No Local Resources** | No local model weights consuming phone RAM or storage |
| **Superior Performance** | Cloud GPUs handle inference instead of Android CPU-only execution |
| **Large Context** | 64k-128k token windows are available |
| **Always Updated** | Hosted model versions can stay current without manual phone-side downloads |
| **Consistent Setup** | The same OCA workflow works across low-end and high-end Android devices |

> **Recommendation**: Use Ollama Cloud Models when you want the Ollama workflow with production-ready cloud performance.

## Official Ollama OpenClaw Integration

OpenClaw integrates with Ollama so you can launch OCA against a selected cloud model without manually editing provider configuration.

### Why It's Useful

1. **Native API Integration**: OpenClaw connects directly to Ollama's native `/api/chat` endpoint for streaming and tool calling.
   > **Important**: Do not use the `/v1` OpenAI-compatible URL with OpenClaw. It can break tool calling and cause models to output raw JSON.
2. **Automatic Model Discovery**: OpenClaw queries `/api/tags` and `/api/show` to discover available Ollama models, detect tool-calling support, and configure context windows.
3. **Android-Friendly Runtime**: Cloud inference keeps the Android device focused on the CLI, gateway, and orchestration work instead of heavy model execution.

### Setup Methods

**Method A: Ollama Launcher (Recommended)**

The easiest way to connect OpenClaw to Ollama is using the official launcher command:

```bash
ollama launch openclaw
```

To launch a specific cloud model directly:

```bash
ollama launch openclaw --model kimi-k2.5:cloud
```

**Method B: OpenClaw Onboarding**

Run the onboarding wizard and select "Ollama" when asked for a provider:

```bash
openclaw onboard
```

It will ask for your Ollama base URL and provider details.

**Method C: Explicit Configuration**

You can force OpenClaw to use Ollama by exporting the API key environment variable before starting the gateway:

```bash
export OLLAMA_API_KEY="ollama-cloud"
openclaw gateway
```

## Cloud Provider Comparison

| Feature | Cloud LLM APIs | Ollama Cloud Models |
|---------|----------------|---------------------|
| **Speed** | Fast, provider-hosted inference | Fast cloud-hosted inference |
| **Device RAM Usage** | Minimal | Minimal |
| **Device Storage Usage** | None for model weights | Minimal registry metadata only |
| **Context Window** | Provider-dependent, often 64k-200k tokens | 64k-128k tokens |
| **Setup** | Configure provider API keys | `ollama pull` and `ollama launch openclaw` |
| **Internet** | Required | Required |
| **Best For** | Production OCA workflows | Ollama-first OCA workflows |

## Troubleshooting

### `ollama launch openclaw` does not show the expected cloud model

Pull the cloud model reference first, then launch again:

```bash
ollama pull kimi-k2.5:cloud
ollama launch openclaw --model kimi-k2.5:cloud
```

### Tool calling returns raw JSON

Make sure OCA is using Ollama's native API route, not the OpenAI-compatible `/v1` endpoint.

### Cloud model requests fail

Check your internet connection, Ollama account/session state, and provider configuration. Cloud-hosted models require network access.

## Best Practices

1. Use cloud-hosted models for all OCA production workflows on Android.
2. Prefer Ollama Cloud Models when you want Ollama's launcher and model selector experience.
3. Keep API keys and provider credentials out of shell history when possible.
4. Run long OCA sessions inside `tmux` to avoid losing work if the terminal disconnects.
