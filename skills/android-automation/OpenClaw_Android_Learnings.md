# OpenClaw on Android — Learnings

Notes from systematic testing of OpenClaw's (Iota's) execution model.
Used to write correct SKILL.md instructions that match actual capabilities.

---

## exec Tool — Complete Behavior Map

| Call | Command duration | Result |
|---|---|---|
| `exec { "command": "..." }` (no yieldMs) | < ~10-15s | Returns stdout inline ✅ |
| `exec { "command": "..." }` (no yieldMs) | > ~10-15s | Returns session ID (default timeout exists) |
| `exec { "command": "...", "yieldMs": N }` | < N ms | Returns stdout inline ✅ |
| `exec { "command": "...", "yieldMs": N }` | > N ms | Returns session ID + PID |

**Key finding:** Plain `exec` without `yieldMs` is NOT truly blocking. It has a default timeout of ~10-15 seconds. Commands longer than that get sessioned regardless.

**Session ID format:** Auto-generated names like "good-cloud", "kind-shoal", "tidy-atlas", "vivid-reef".

---

## process Tool — Complete Behavior Map

| Call | Timing | Result |
|---|---|---|
| `process { "action": "poll", "sessionId": "..." }` | While running | `"(no new output)\nProcess still running."` — returns immediately |
| `process { "action": "poll", "sessionId": "..." }` | After finished | Nothing / empty |
| `process { "action": "log", "sessionId": "...", "limit": 50 }` | While running | **Causes OpenClaw turn to timeout** ☠️ |
| `process { "action": "log", "sessionId": "...", "limit": 50 }` | After finished | Nothing |

**Key finding:** Neither `poll` nor `log` can retrieve stdout from a long-running or completed command. `process/log` while a command is running causes OpenClaw's entire turn to timeout — it must never be used during automation.

---

## What This Means

**There is no mechanism to get stdout from any command that runs longer than ~10-15 seconds.**

The ONLY bridge between a background process and OpenClaw is the **filesystem**:
- `~/storage/shared/android_agent/last_result.json` — written by `run.py` when automation finishes
- `~/android-automation-agent/scripts/check_busy.sh` — reads the lock file, returns `FREE` or `BUSY: ...`

## Async Notification Mechanisms

| Mechanism | Works? | Notes |
|---|---|---|
| `notifyOnExit` callback | ❌ No | Does not reliably fire |
| `sleep` + TTS | ❌ Unreliable | Sometimes works, not consistent |
| Cron jobs | ✅ Yes | Only reliable async mechanism in OpenClaw |
| Telegram (from Python) | ✅ Yes | `run.py` + `runner.py` handle this automatically |

**Implication:** Do not design any flow that depends on callbacks or post-command notifications from OpenClaw. The automation itself sends Telegram updates every 2 steps and a final result — that is the notification channel.

## Correct Pattern for Android Automation

### Start (non-blocking)
```
exec { "command": "cd ~/android-automation-agent && python run.py '<GOAL>' --steps <N> --json", "yieldMs": 5000 }
```

- `yieldMs: 5000` gives a 5-second window for instant tasks to return inline
- For all real automation (30s–5min), gets session ID → runs in background
- Tell user: "Started. Watch Telegram for live updates."
- OpenClaw stays **responsive**

### Get result (when user comes back)
```
exec { "command": "bash ~/android-automation-agent/scripts/check_busy.sh" }
```
- `FREE` → read result: `exec { "command": "cat ~/storage/shared/android_agent/last_result.json" }`
- `BUSY` → "Still running, watch Telegram"

### Never do
- `process/log` — causes turn timeout
- `process/poll` in a loop — returns nothing, wastes turns
- Check `last_result.json` while automation is running — file not yet written
- Use named sessions manually — they're auto-generated by the system

---

## Why the Infinite Loop Happened

1. OpenClaw used `exec + yieldMs` (small default) → got session ID "good-cloud"
2. Tried `process/log` to read session output → got nothing (or timed out)
3. Couldn't find the result → retried → looped
4. Each loop: "Starting now... 👾 / Checking session good-cloud... 👾"
5. Infinite loop until user interrupted

Root fix: stop trying to read session output. Use `check_busy.sh` + `last_result.json` instead.

## How Telegram Updates Work (No OpenClaw Action Needed)

The Python automation handles all Telegram notifications automatically:

1. **Start**: `run.py` sends "🤖 Android Agent started" before first ADB action
2. **Screen ready**: `run.py` sends screenshot after wake/unlock
3. **Every 2 steps**: `runner.py` sends screenshot + caption (step count, subgoal, last action)
4. **Final result**: `runner.py` `finally` block sends result even on crash/cancel

OpenClaw does not need to send any Telegram messages. It just starts the command and reads `last_result.json` when the user comes back.
